📦 Installation · Standalone

Pour quels cas d'usage voudrait-on installer une stack Elastic complète sur un seul serveur ?

Cas valides :
  • Environnement de développement / lab : tester des pipelines Logstash, prototyper des dashboards Kibana sans infrastructure.
  • Petite organisation : centraliser des logs d'un seul service web, volume limité (quelques Go/jour).
  • POC & démo : valider l'architecture avant un cluster de production.
Pas adapté à : haute disponibilité, tolérance aux pannes, gros volumes, résilience.
Dev / LabPas pour la prod HA
📋 Objectif · Définition

Qu'est-ce que l'analyse des logs ?

L'analyse de logs est le processus consistant à donner du sens aux messages de logs générés par un système, dans le but d'améliorer les performances ou résoudre des problèmes.

Elle permet de : détecter des anomalies, recréer une chaîne d'événements, surveiller la sécurité, analyser le comportement des utilisateurs et se conformer aux réglementations (RGPD, PCI-DSS…).
Fondamental
❓ Analyse de logs

Pourquoi l'analyse des logs est-elle importante ?

  • Conformité légale : obligation d'archiver et analyser les logs (RGPD, PCI-DSS, ISO 27001…)
  • Détection d'incidents : détecter les erreurs HTTP, les tentatives d'intrusion, les anomalies
  • Résolution de problèmes : recréer la chaîne d'événements ayant conduit à un incident
  • Performance : identifier les bottlenecks avant qu'ils impactent les utilisateurs
  • Gain de temps : intervenir rapidement plutôt que de dépanner manuellement
✅ Bonnes pratiques

Quelles sont les meilleures pratiques d'analyse des logs ?

  • Détection de patterns : filtrer les messages entrants via expressions régulières pour détecter rapidement les anomalies
  • Normalisation : convertir IP, timestamps dans un format commun à toutes les équipes
  • Classification / balisage : tagger les messages avec des mots-clés, les catégoriser en classes
  • Analyse de corrélation : collecter des données de plusieurs sources pour relier les événements
  • Alerte : déclencher des notifications quand un pattern anormal est détecté
⚙️ JVM · Mémoire

Que signifient les paramètres -Xms512m et -Xmx512m ?

-Xms (Xmemory start) = taille initiale du heap JVM alloué au démarrage.
-Xmx (Xmemory max) = taille maximale que le heap peut atteindre.

Les mettre égaux (512m) évite les réallocations dynamiques coûteuses lors de l'exécution. En production, on recommande ~50% de la RAM disponible, maximum 32 Go (au-delà, les pointeurs JVM deviennent 64-bit et la GC se dégrade).
-Xms = initial · -Xmx = max
🖥️ Système · ps

Comment se lit la sortie de la commande ps ?

ps auxf : a = tous les utilisateurs, u = format détaillé (USER, PID, %CPU, %MEM, VSZ, RSS, STAT, START, TIME, COMMAND), x = inclure les processus sans terminal (daemons), f = forest (arbre de processus avec indentation).

Colonnes importantes : PID = identifiant du processus, %CPU/%MEM = consommation, VSZ = mémoire virtuelle, RSS = mémoire physique réelle, STAT = état (S=sleeping, R=running, Z=zombie…).
🌐 Réseau · ss

Qu'est-ce que la commande ss ?

ss (Socket Statistics) = remplaçant moderne de netstat. Plus rapide car lit directement depuis le kernel via Netlink.

Options courantes : -l listening (ports en écoute), -n numeric (pas de résolution DNS/nom), -t TCP uniquement, -u UDP, -p affiche le processus propriétaire, -a tous les états.

ss -lntp → tous les ports TCP en écoute avec le PID du processus.
🔐 Sécurité · Auth

Quelles sont les différentes identités utilisées dans la commande curl pour tester Elasticsearch ?

3 niveaux d'identité :
  • sudo -u elasticsearch → identité OS : exécute curl en tant qu'utilisateur système elasticsearch pour avoir accès au certificat TLS dans /etc/elasticsearch/certs/
  • --cacert .../http_ca.crt → identité TLS : certificat de l'autorité de certification pour valider le serveur
  • -u elastic → identité applicative : compte utilisateur Elasticsearch (superuser) avec son mot de passe
OS → TLS → Appli
🚀 Service · Lifecycle

Pourquoi n'a-t-on pas activé le démarrage automatique du service dès l'installation ?

Parce qu'on doit d'abord configurer le service (mémoire JVM, sécurité TLS, enrollment token…) avant qu'il démarre automatiquement.

Activer systemctl enable trop tôt risque :
  • De démarrer un service mal configuré au prochain reboot
  • De générer des boucles de redémarrage en échec qui corrompent les données
  • D'exposer un service non sécurisé sur le réseau
Bonne pratique : Configurer → Tester → Enable → Reboot de validation.
Configure → Enable → Reboot
🛡️ SSH · Port Forwarding

Pourquoi utiliser une redirection de port vers votre hôte plutôt qu'installer un navigateur sur le serveur ?

  • OS minimal sans GUI : pas de desktop ni navigateur graphique installé (économie de RAM, surface d'attaque réduite)
  • Sécurité : Kibana écoute sur 127.0.0.1:5601 uniquement, non exposé sur le réseau
  • Chiffrement : le tunnel SSH chiffre le trafic HTTP de Kibana sans ouvrir de port supplémentaire dans le firewall
ssh -L 127.0.0.1:15601:127.0.0.1:5601 = redirige le port local 15601 du poste de travail vers le port 5601 du serveur via SSH.
🛡️ SSH · Syntaxe

Comment interpréter la commande SSH de tunnel ssh -L 127.0.0.1:15601:127.0.0.1:5601 -l user -p PORT IP ?

  • -L 127.0.0.1:15601:127.0.0.1:5601 → Local forward : tout trafic arrivant sur localhost:15601 du poste de travail est transmis vers localhost:5601 du serveur distant via le tunnel
  • -l user → login : nom d'utilisateur SSH
  • -p PORT → port SSH du serveur (si différent de 22)
  • -v → verbose : affiche les détails de la connexion (utile pour debug)
Format général : ssh -L [bind_addr:]local_port:remote_host:remote_port
🛠️ Commande · sed

Comment interpréter la commande sed 's/#--max-old-space-size=4096/--max-old-space-size=1024/' -i /etc/kibana/node.options ?

sed = Stream EDitor, traite le texte ligne par ligne.
s/PATTERN/REMPLACEMENT/ = substitution (remplace la 1re occurrence par ligne).
  • #--max-old-space-size=4096 = pattern à chercher (ligne commentée avec valeur 4096 Mo)
  • --max-old-space-size=1024 = remplacement (dé-commente et réduit à 1024 Mo)
  • -i = in-place : modifie le fichier directement (sans -i, la sortie va sur stdout)
Résultat : la mémoire Node.js de Kibana passe de 4 Go à 1 Go (adaptation au lab avec peu de RAM).
🔄 Logstash · ETL

Quel nom générique donne-t-on aux produits qui ont les fonctionnalités de Logstash (Extract, Transform, Load) ?

On les appelle des outils ETL (Extract, Transform, Load) ou parfois ELT (Extract, Load, Transform) dans le cas du Big Data.

Logstash implémente l'ETL pour les données de logs :
  • Extract : plugins input (file, beats, kafka, syslog, http…)
  • Transform : plugins filter (grok, mutate, date, json, geoip…)
  • Load : plugins output (elasticsearch, s3, kafka, email, stdout…)
Autres ETL du marché : Apache NiFi, Talend, Informatica, AWS Glue, Airbyte.
ETL = Extract → Transform → Load
📈 Monitoring · vmstat

À quoi sert vmstat -S M 60 ? Quelles informations donne-t-elle ?

vmstat (Virtual Memory Statistics) affiche toutes les 60 secondes :
  • procs : r = processus en attente de CPU, b = processus bloqués en I/O
  • memory : swpd (swap utilisé), free, buff, cache (en Mo avec -S M)
  • swap : si (swap in/s), so (swap out/s) — si >0 : mémoire insuffisante !
  • io : bi (blocs lus/s), bo (blocs écrits/s)
  • system : in (interruptions/s), cs (context switches/s)
  • cpu : us (user%), sy (kernel%), id (idle%), wa (I/O wait%)
💻 Shell · Scripting

Comment fonctionne le code qui préfixe les lignes de vmstat avec un timestamp ?

vmstat -S M 60 | (while read line ; do echo -e "$(date '+%Y-%m-%d %H:%M:%S %z')|$line"; done)

  • | pipe : la sortie de vmstat est envoyée ligne par ligne à la boucle
  • while read line : lit chaque ligne de vmstat dans la variable $line
  • $(date '…') : substitution de commande — exécute date à chaque itération et insère la date courante
  • echo -e "…|$line" : affiche date + séparateur + ligne vmstat
Le -e de echo active l'interprétation des séquences d'échappement (\n, etc.).
🕒 NTP · Synchronisation

Pourquoi s'inquiéter de mettre notre serveur à l'heure ?

  • Corrélation de logs : des timestamps incorrects rendent impossibles les analyses temporelles dans Kibana (un log "avant" peut apparaître "après")
  • Sécurité : les certificats TLS ont des dates de validité — un décalage d'horloge peut invalider des certificats valides
  • Cluster Elasticsearch : les nœuds d'un cluster doivent être synchronisés (écart toléré < 30s)
  • Débogage : sans horloge synchronisée, impossble de corréler des événements entre différents systèmes
Commandes utiles : sudo systemctl start systemd-timesyncd, sudo timedatectl timesync-status
📊 Monitoring · Charge

Comment savoir si le système d'exploitation qui porte les services est chargé ?

  • uptime ou cat /proc/loadavg : load average sur 1/5/15 min. Si > nb de vCPU → saturé
  • top / htop : vue dynamique CPU, RAM, swap par processus
  • vmstat 1 5 : statistiques système toutes les secondes
  • iostat -x 1 : saturation I/O disque (iowait%, util%)
  • free -h : mémoire disponible/utilisée/cache
Règle : load average 1min > nb CPUs = surcharge courante, 15min > nb CPUs = problème structurel.
💻 Shell · One-liner

Expliquer le "one-liner" de vérification des services ELK

for servicename in elasticsearch kibana logstash ; do → boucle sur les 3 services
date ; echo checking $servicename ... → affiche heure + nom du service vérifié

📝 Logstash · Formats

Qu'est-ce qu'un log « structuré » ?

Un log structuré suit un format prédéfini et parsable automatiquement, avec des champs nommés à des positions prévisibles ou avec des séparateurs clairs.

Exemples structurés : Apache Combined Log (IP date méthode code taille user-agent), syslog, JSON logs, CSV.

Exemples non structurés : messages d'erreur texte libre d'une application maison.

Grok peut extraire des champs même depuis des logs semi-structurés grâce aux expressions régulières nommées. Le pattern COMBINEDAPACHELOG est prédéfini.
🔎 Grok · Apache

Que reconnaît-on dans le message de log Apache traité par Logstash ?

Le pattern COMBINEDAPACHELOG décompose 127.0.0.1 - - [17/Jan/2026:17:38:02 +0100] "GET / HTTP/1.1" 200 10956 "-" "curl/7.88.1" en :
  • source.address = 127.0.0.1
  • @timestamp = 2026-01-17T16:38:02.000Z (converti par le plugin date)
  • http.request.method = GET
  • url.original = /
  • http.version = HTTP/1.1
  • http.response.status_code = 200 (integer après mutate)
  • http.response.body.bytes = 10956 (integer)
  • user_agent.original = curl/7.88.1
🎛️ API · Méthodes

Dans quels cas préférera-t-on l'API curl vs Kibana Dev Tools ?

API curl / ligne de commande : scripting et automatisation (CI/CD, cron), accès sans interface graphique (session SSH), chargement bulk de données, intégration dans des scripts bash/Python, monitoring automatisé.

Kibana Dev Tools (Console) : exploration interactive, auto-complétion des requêtes DSL Elasticsearch, visualisation immédiate des résultats formatés, débogage de requêtes complexes, documentation intégrée.

Règle pratique : si c'est répétable → script curl. Si c'est exploratoire → Kibana Dev Tools.
📄 Elasticsearch · Pagination

Pourquoi ne peut-on pas récupérer tous les documents d'un coup ? Qu'est-ce que la pagination Elasticsearch ?

Elasticsearch limite par défaut les résultats à 10 000 documents max par requête. Raison : scanner massivement sature le heap JVM et dégrade les performances pour tous les utilisateurs.

Solutions de pagination :
  • from + size : simple mais inefficace pour de grandes pages (relit depuis le début)
  • search_after : utilise la valeur du dernier doc comme curseur → recommandé en production
  • scroll API : crée un snapshot de l'index → pour exports massifs (déprécié en faveur de search_after)
  • Point In Time (PIT) : snapshot + search_after → approche moderne
📊 Volume · Big Data

114 000 documents, c'est beaucoup ?

Non, c'est très petit à l'échelle d'Elasticsearch/Kibana.

Pour référence : un serveur web moyen génère plusieurs millions de lignes de log par jour. Les clusters de production Elasticsearch stockent souvent des milliards de documents sur des pétaoctets.

114 000 docs tient entièrement en RAM, les recherches sont quasi-instantanées. Le challenge begin à partir de quelques dizaines de millions de documents (gestion de la mémoire, sharding, ILM).
📊 Kibana · Dashboard

Quelles visualisations seraient utiles dans un dashboard Apache ?

  • Line / Bar chart : volume de requêtes par intervalle de temps (détection de pics)
  • Pie / Donut : répartition des codes HTTP (200/304/404/500) ou URLs les + visitées
  • Data Table : Top 10 URLs, Top 10 user-agents, octets moyens par requête
  • Metric : taux d'erreur total, nombre total de requêtes, bytes servis
  • Map : géolocalisation des IPs sources (plugin GeoIP Logstash)
  • TSVB : évolution du temps de réponse ou du code HTTP dans le temps

ELK Installation Elasticsearch

# Installer Elasticsearch (version précise pour le lab)
user@nosql01:~$ sudo apt-get install elasticsearch=9.2.4

# Configurer la JVM (réduire heap à 512Mo pour le lab)
user@nosql01:~$ sudo nano /etc/elasticsearch/jvm.options

-Xmx512m  # heap max = 512 Mo (égaux = pas de réallocation)

# Démarrer et vérifier le service
user@nosql01:~$ sudo systemctl daemon-reload
user@nosql01:~$ sudo systemctl start elasticsearch

# Tester l'API (avec cert TLS et authentification)
user@nosql01:~$ sudo -u elasticsearch curl --cacert /etc/elasticsearch/certs/http_ca.crt -u elastic https://localhost:9200

# Passer en démarrage automatique APRÈS configuration
user@nosql01:~$ sudo systemctl enable elasticsearch
Pourquoi sudo -u elasticsearch ? Pour exécuter curl avec les droits de l'utilisateur système elasticsearch qui a accès au certificat dans /etc/elasticsearch/certs/.

user@nosql01:~$ sudo -u logstash /usr/share/logstash/bin/logstash --path.settings /etc/logstash -f /etc/logstash/conf.d/apache.conf --config.test_and_exit

# Lancer en mode debug pour voir les événements traités
user@nosql01:~$ sudo -u logstash /usr/share/logstash/bin/logstash --path.settings /etc/logstash -f /etc/logstash/conf.d/apache.conf --debug

# Générer des logs Apache de test
user@nosql01:~$ curl localhost:80
user@nosql01:~$ links http://127.0.0.1:80/manual  # génère un 404
user@nosql01:~$ wget -o /dev/null http://127.0.0.1:80

# Script de génération massive de logs (1 à 20 req, 4 URLs, 2 clients)
user@nosql01:~$ export maxrq=20 ; for client in 'wget -o /dev/null' 'curl -o /dev/null' ; do for url in 'http://127.0.0.1:80' 'http://127.0.0.1/manual' 'http://127.0.0.1/absent' ; do for i in $(seq 1 $((1 + $RANDOM % $maxrq))) ; do $client $url ; done ; done ; done
ELK Configuration Logstash Pipeline (apache.conf)
logstash config · /etc/logstash/conf.d/apache.conf
# INPUT : surveille le fichier de logs Apache en temps réel
input {
  file { path => "/var/log/apache2/access.log" }
}

# FILTER : parse, convertit et horodate
filter {
  grok {
    # Extrait tous les champs du log Apache Combined
    match => { "message" => "%{COMBINEDAPACHELOG}" }
  }
  date {
    # Remplace @timestamp par la date du log (pas la date de traitement)
    match => [ "timestamp" , "dd/MMM/yyyy:HH:mm:ss Z" ]
  }
  mutate {
    # Convertit les champs texte en entiers pour les agrégations Kibana
    convert => { "response" => "integer"  "bytes" => "integer" }
  }
}

# OUTPUT vers Elasticsearch avec authentification API key
output {
  elasticsearch {
    hosts => "https://localhost:9200"
    ssl_certificate_authorities => "/etc/logstash/config/certs/http_ca.crt"
    ssl_enabled => true
    api_key => "<id>:<api-key>"
    # Index par jour → facilite la gestion du cycle de vie (ILM)
    index => "apache-%{+YYYY.MM.dd}"
  }
}
ELK API Elasticsearch — CRUD complet
bash · Elasticsearch REST API
# Variable pour la clé API (évite de la laisser dans l'historique)
user@nosql01:~$ export ESKEY="cWxMQ...Zw=="

# Lister les index
user@nosql01:~$ sudo -u elasticsearch curl -X GET -H "Authorization: ApiKey $ESKEY" --cacert /etc/elasticsearch/certs/http_ca.crt "https://localhost:9200/_cat/indices?v"

# Insérer un document (crée l'index "mesrecettes" au passage)
user@nosql01:~$ sudo -u elasticsearch curl -X POST -H "Authorization: ApiKey $ESKEY" --cacert ... "https://localhost:9200/mesrecettes/_doc?pretty" -H 'Content-Type: application/json' -d '{"title":"Mon canard aux oranges","category":"Plats","author":"Moi"}'

# Insertion bulk (plusieurs documents en une requête)
user@nosql01:~$ ... curl -X POST ... "https://localhost:9200/mesrecettes/_bulk?pretty" ... -d '
{ "create": {} }
{ "title": "La dinde au whisky", "category": "Plats", "author": "Tatie Josette" }
{ "create": {} }
{ "title": "Le cake au thon", "category": "Entrées", "author": "Tatie Josette" }
'

# Recherche : tous les docs
user@nosql01:~$ ... curl -X GET ... "https://localhost:9200/mesrecettes/_search?pretty" -d '{"query":{"match_all":{}}}'

# Recherche avancée : plats PAS de "Moi" (must + must_not)
user@nosql01:~$ ... -d '{"query":{"bool":{"must":[{"match":{"category":"Plats"}}],"must_not":[{"match":{"author":"Moi"}}]}}}'

# Recherche : phrase exacte "LE BOURG" avec filtre et taille
user@nosql01:~$ ... ".../_search?pretty&filter_path=hits.total,hits.hits._source&size=20" -d '{"query":{"match_phrase":{"lb_voie_ext":"LE BOURG"}}}'

# Modifier un document existant (ajouter un champ)
user@nosql01:~$ ... curl -X POST ... "https://localhost:9200/mesrecettes/_update/ID_DU_DOC?pretty" -d '{"doc":{"score":3}}'

# Supprimer un document
user@nosql01:~$ ... curl -X DELETE ... "https://localhost:9200/indexnul/_doc/ID?pretty"

# Supprimer un index entier
user@nosql01:~$ ... curl -X DELETE ... "https://localhost:9200/indexnul?pretty"

📦 Installation

Pour quels cas d'usage voudrait-on installer un MongoDB standalone ?

  • Développement / prototypage : une seule instance suffit, setup rapide
  • Tests unitaires : base isolée par test
  • Petites applications : volume faible, pas de besoin de résilience
  • Apprentissage : comprendre MongoDB avant un cluster
Pas adapté : haute disponibilité, tolérance aux pannes, gros volumes → utiliser replica set minimum.
🖥️ OS · VM

Pourquoi ne pas installer de desktop ? Pourquoi pas de mises à jour automatiques ?

Pas de desktop : réduit la surface d'attaque (moins de packages = moins de vulnérabilités), économise la RAM/CPU pour MongoDB, reflète les bonnes pratiques de production (serveurs headless).

Pas de MAJ auto : une mise à jour MongoDB peut introduire des régressions ou changements incompatibles. En production, les MAJ doivent être planifiées, testées, et appliquées en fenêtre de maintenance avec validation du DBA.
🔒 Version pinning

À quoi sert la commande echo "mongodb-org hold" | sudo dpkg --set-selections ?

Cette commande gèle la version du paquet MongoDB installé. dpkg --set-selections avec l'état hold empêche apt d'upgrader ce paquet lors des apt upgrade.

Utile pour : éviter des régressions fonctionnelles en cours de lab, reproduire un environnement de production figé, attendre une validation avant une montée de version.

Vérifier : dpkg --get-selections | grep hold. Enlever le gel : echo "mongodb-org install" | sudo dpkg --set-selections
dpkg hold = version gelée
🔐 Auth · Sécurité

Pourquoi le message "Access control is not enabled" alors qu'on vient de configurer la sécurité ?

La configuration security: authorization: enabled a été écrite dans /etc/mongod.conf, mais le service n'a pas encore été redémarré. MongoDB lit sa configuration uniquement au démarrage.

Le processus en cours d'exécution tourne toujours avec l'ancienne configuration (sans auth). C'est pourquoi on crée l'utilisateur admin avant de redémarrer : si on activait d'abord l'auth, personne ne pourrait plus se connecter pour créer cet admin.
⚠️ Auth · Risques

Qu'est-ce qui pourrait mal se passer dans la procédure d'activation de l'authentification ?

Scénario catastrophe : activer authorization: enabled et redémarrer AVANT de créer le compte admin → lock-out total, plus personne ne peut se connecter.

Solution d'urgence : la "localhost exception" — si aucun utilisateur n'existe dans la base admin, une connexion locale sans auth est autorisée une seule fois.

Ordre correct :
  1. Démarrer MongoDB sans auth (défaut)
  2. Créer le compte admin
  3. Activer authorization: enabled dans mongod.conf
  4. Redémarrer MongoDB
  5. Vérifier la connexion avec le compte admin
Créer admin AVANT d'activer auth
🗄️ Base · Lazy creation

Pourquoi ne voit-on pas la DB "user-data" dans listDatabases après use user-data ?

MongoDB utilise une création paresseuse (lazy creation) : la commande use mydb bascule simplement le contexte côté client, sans créer physiquement de base de données sur le disque.

La base n'apparaît dans listDatabases qu'après qu'elle contient au moins une collection avec des données. db.createCollection() crée la collection (et donc la base) même sans données.

🔑 Sécurité · Intégrité

Que fait sha256sum ? À quoi cela sert-il ?

sha256sum fichier calcule l'empreinte cryptographique (hash SHA-256) d'un fichier.

Utilité : vérifier l'intégrité du fichier téléchargé. Si le hash correspond à celui publié par l'éditeur sur son site officiel, le fichier n'est ni corrompu (erreur réseau) ni falsifié (attaque man-in-the-middle).

SHA-256 produit une empreinte de 64 caractères hexadécimaux. Une modification d'un seul bit du fichier change complètement l'empreinte.

📌 Index · Exécution

Commentez la création d'index et son effet sur le plan d'exécution

Avant index : stage: 'COLLSCAN' = scan complet de toute la collection (O(n) documents parcourus).

Après db.restaurants.createIndex({"address.zipcode": 1}) :
IXSCAN (Index Scan) → FETCH : MongoDB parcourt l'index B-tree (O(log n)), puis récupère uniquement les documents matchants.

Le 1 = ordre croissant (-1 = décroissant). L'impact sur les performances est significatif sur de grandes collections mais peu visible sur 25 000 docs (tout en RAM).
☁️ Architecture · Conteneurs

Qu'est-ce qu'une application stateful / stateless ? Quelles contraintes avec les conteneurs ?

Stateless : ne stocke aucun état entre les requêtes. Chaque instance est identique et interchangeable. Facile à scaler horizontalement (ex: API REST).

Stateful : maintient un état persistant (données, sessions). Chaque instance a une identité propre (ex: MongoDB, Redis).

Contraintes conteneurs + stateful :
  • Les données doivent être dans des volumes persistants (PersistentVolume en Kubernetes)
  • Le scheduler doit respecter l'affinité nœud/données (ne pas déplacer un pod sans ses données)
  • Les StatefulSets Kubernetes gèrent les noms de pod stables et le démarrage ordonné
  • Attention aux race conditions lors des redémarrages simultanés
🗺️ Cluster · Topologie

Qu'entend-on par "proche" pour les routers MongoDB ?

"Proche" signifie faible latence réseau entre le client et le router (mongos).

En pratique : le router est déployé sur le même datacenter voire le même rack que les applications clientes, ou sur des serveurs d'application dédiés (co-location).

Un router distant (autre continent) introduirait une latence inacceptable pour chaque requête. En microservices, on déploie souvent un mongos par zone de disponibilité (AZ).
⚠️ Architecture · SPOF

Qu'est-ce qu'un SPOF ? Pourquoi distribuer les primaires sur des serveurs différents ?

SPOF (Single Point of Failure) = composant dont la panne arrête tout le système. Tout système de production doit éliminer ses SPOF.

Primaires sur serveurs différents : si le serveur 1 tombe, seul le shard1 perd son primary (un secondaire est élu). Les autres shards continuent normalement. Si tous les primaires étaient sur le même serveur, sa panne stopperait toutes les écritures.

En production : distribuer aussi sur des zones de disponibilité différentes (AZ), voire des régions géographiques.
🌐 DNS · Nomenclature

Qu'est-ce qu'un FQDN ? Pourquoi reprendre la nomenclature d'un cluster complexe pour un cluster simple ?

FQDN (Fully Qualified Domain Name) = nom DNS complet incluant tous les niveaux de la hiérarchie. Ex : mongo-shard1-r1.lab.local = hostname + domaine + TLD.

Pourquoi reprendre la nomenclature complexe :
  • La configuration reste identique que ce soit sur 1 ou 3 serveurs physiques
  • Facilite la migration vers un vrai cluster (juste changer les entrées DNS)
  • Bonne pratique d'architecture : nommer par rôle (shard1-r1) pas par serveur physique
🔌 Réseau · Loopback

Que sont les adresses 127.0.3.x ? Pourquoi les utiliser dans ce lab ?

Les adresses 127.0.0.0/8 sont toutes des adresses de loopback (interface lo) sur Linux. Seule 127.0.0.1 est conventionnellement utilisée, mais toute la plage est valide.

Pourquoi 127.0.3.x dans le lab :
  • Simuler des "serveurs différents" sur une seule machine physique
  • Chaque service écoute sur une adresse distincte → isolation des ports possible
  • Pas besoin d'interfaces réseau réelles ou de VMs supplémentaires
  • Le trafic reste en loopback = rapide, pas de problème réseau

🌐 DNS · OS

Pourquoi utiliser /etc/hosts dans ce lab ? Mécanismes de résolution de noms sur Linux/Windows ?

/etc/hosts = résolution locale sans DNS server. Parfait pour un lab sur une seule machine.

Linux : l'ordre est défini par /etc/nsswitch.conf (hosts: files dns). D'abord /etc/hosts, ensuite le DNS configuré dans /etc/resolv.conf.

⚙️ OS · Services

Qu'est-ce qu'un service ? Différence avec un programme ? Un processus ?

  • Programme = fichier exécutable sur disque (code binaire compilé ou script). Ex : /usr/bin/mongod
  • Processus = instance en cours d'exécution d'un programme. A un PID, utilise de la RAM et du CPU, peut être en sleep, en attente I/O, en exécution…
  • Service / daemon = processus géré par le gestionnaire de services (systemd) : démarre automatiquement au boot, redémarre en cas d'échec, a un journal centralisé (journald), des dépendances, une description.
systemd orchestre les services. systemctl start lance le processus, systemctl enable le met en démarrage automatique.
🔧 systemd · systemctl

À quoi sert systemctl ? Pourquoi exécuter systemctl status après un démarrage ?

systemctl = interface de contrôle du gestionnaire de services systemd.
  • start/stop/restart : démarre/arrête/redémarre un service
  • enable/disable : active/désactive le démarrage automatique
  • status : affiche l'état, les dernières lignes de log, le PID
  • daemon-reload : recharge les fichiers .service modifiés
Pourquoi status après démarrage : vérifier que le service est vraiment actif (Active: active (running)) et non en échec silencieux. Les dernières lignes du log permettent aussi de détecter des erreurs au démarrage.
⏱️ Shell · Timing

Pourquoi insérer sleep 2 entre les deux commandes systemctl restart et systemctl status ?

MongoDB prend quelques secondes pour s'initialiser après un restart. Sans délai, systemctl status pourrait afficher activating (start) au lieu de active (running), donnant une fausse impression.

sleep 2 attend 2 secondes pour laisser le temps au service de démarrer et d'atteindre un état stable avant de vérifier son statut. C'est une validation simple mais efficace pour les scripts de déploiement.
🔍 Processus · mongod

Pourquoi y a-t-il 2 processus mongod ? Pourquoi les 2 n'écoutent pas sur la même adresse ?

2 processus mongod :
  • Le premier (/etc/mongod.conf) = l'instance standalone installée au début du TP (port 27017 sur 127.0.0.1)
  • Le second = le config server du cluster (port 28101 sur mongo-config-r1 = 127.0.3.101)
Adresses différentes : chaque service est lié (--bind_ip) à son alias réseau spécifique. Cela évite les conflits de ports (deux mongod ne peuvent pas écouter sur le même IP:port) et isole les services logiquement.
💻 Shell · grep

À quoi sert le grep -v grep dans la commande ps aux | grep -v grep | grep mongo ?

Quand on lance grep mongo, ce processus lui-même apparaît dans la liste ps avec "grep mongo" dans sa ligne de commande.

grep -v grep exclut (-v = invert) toutes les lignes contenant "grep", éliminant ce faux positif.

Alternative moderne : pgrep -a mongod ou ps aux | grep [m]ongo (l'astuce des crochets : [m]ongo ne se match pas lui-même car le pattern n'est pas "mongo").
⚙️ Service · ExecStart

À quoi sert la ligne ExecStart du fichier de service ? Détaillez chaque argument pour mongo-config-r1

/usr/bin/mongod --configsvr --port 28101 --replSet rs0 --dbpath /srv/data/mongo-config-r1 --bind_ip mongo-config-r1 --logpath=/var/log/mongodb/mongo-config-r1.log
  • /usr/bin/mongod = binaire MongoDB (daemon)
  • --configsvr = rôle "config server" (stocke métadonnées du cluster)
  • --port 28101 = port d'écoute (≠ 27017 par défaut)
  • --replSet rs0 = nom du replica set des config servers
  • --dbpath /srv/data/mongo-config-r1 = dossier de stockage des données
  • --bind_ip mongo-config-r1 = n'écoute QUE sur cet alias réseau
  • --logpath=... = fichier de log dédié
🔄 Replica Set · Status

Expliquez les éléments importants dans la sortie de rs.status()

  • set : nom du replica set (rs0 ou shard1…)
  • myState : état du nœud actuel (1=PRIMARY, 2=SECONDARY, 6=UNKNOWN/STARTUP)
  • members[].stateStr : PRIMARY, SECONDARY, RECOVERING…
  • members[].health : 1=accessible, 0=inaccessible
  • members[].syncSourceHost : depuis quel nœud ce secondaire se synchronise
  • members[].pingMs : latence inter-nœuds
  • electionDate : date de la dernière élection du primary
  • majorityVoteCount : nombre de votes nécessaires pour élire un primary
🔐 Cluster · Sécurité

Que nous dit le message "Access control is not enabled" sur le router ? Devrait-on s'en inquiéter ?

Oui, en production on devrait absolument s'en inquiéter. Ce message signifie que n'importe quel client peut se connecter au cluster sans authentification et lire/modifier/supprimer toutes les données.

Dans ce lab, on l'ignore délibérément pour simplifier le setup du cluster. En production, il faut :
1. Activer l'authentification interne entre nœuds (keyFile ou x509)
2. Créer des utilisateurs MongoDB sur le cluster
3. Activer security.authorization sur tous les mongod/mongos
Référence : MongoDB Security Checklist.

🚀 Service · Enable

Pourquoi passer les services en démarrage automatique ? Et pourquoi attendre la fin de la création ?

Pourquoi enable : après un reboot (planned ou crash), les services doivent redémarrer automatiquement sans intervention manuelle. Indispensable en production.

Pourquoi attendre la fin : activer le démarrage auto d'un service mal configuré provoquera des boucles de redémarrage en échec à chaque boot. On active uniquement quand le service est validé comme fonctionnel.
💻 Shell · Brace expansion

Qu'est-ce que la syntaxe {mongo-config-r1,mongo-shard1-r1,mongo-router1} ?

C'est la brace expansion (expansion d'accolades) du shell bash. Elle génère plusieurs mots à partir d'un préfixe/suffixe commun.

sudo systemctl enable {mongo-config-r1,mongo-shard1-r1,mongo-router1}
est équivalent à :
sudo systemctl enable mongo-config-r1 mongo-shard1-r1 mongo-router1

Très utile pour éviter la répétition. Fonctionne aussi avec des séquences : {1..10}, {a..z}.
💻 Shell · Prompt

Qu'appelle-t-on "prompt" dans un shell ? Quel est l'intérêt de le capturer ?

Le prompt = l'invite de commande affichée par le shell avant chaque saisie. Ex : user@nosql010:~$.

Il indique typiquement : utilisateur, hostname, répertoire courant, et parfois le contexte (branch git, env virtuel Python, rôle MongoDB comme shard1 [direct: primary]).

Intérêt de le capturer dans une copie d'écran : prouve sur quelle machine et sous quel utilisateur la commande a été exécutée, et quel était l'état du shell au moment de l'exécution. Indispensable pour la documentation et le débogage.

📊 Performance · Maquette

Quelles sont les difficultés qui s'opposent à l'évaluation des performances sur cette maquette ?

  • Contention CPU/RAM : tous les services (7+ mongod, mongos, Kibana, ES, Logstash) partagent les ressources d'un seul OS
  • Réseau loopback : latence ~0ms entre nœuds, non représentatif d'un vrai réseau (10Gbps datacenter = ~1ms, WAN = 10-100ms)
  • Cache OS surchargé : données des tests précédents restent en cache → temps artificialmente courts
  • Pas de charge concurrente : un seul client de test, pas de trafic réaliste multi-utilisateurs
  • Pas de réplication réelle : replica sets à 1 nœud = pas de surcoût de réplication
🚀 Cluster · Évolution

Que faudrait-il faire pour augmenter la capacité du cluster ? Quels axes d'amélioration ?

Capacité données : ajouter un 3ème shard (sh.addShard("shard3/mongo-shard3-r1:28131")) → MongoDB rééquilibre automatiquement les chunks (balancer).

Résilience : minimum 3 membres par replica set (1 primary + 2 secondary) distribués sur des serveurs/AZ différents. 3 config servers (déjà en place).

Performance lecture : router vers les secondaires (readPreference: secondaryPreferred) pour distribuer la charge.

Shard key : choisir une clé à haute cardinalité et distribution uniforme. Éviter les hot spots (ObjectId croissant sans hashing). Analyser avec db.collection.getShardDistribution().

Sécurité : activer TLS inter-nœuds, authentification par keyFile ou x509, segmentation réseau.

Automatisation : Kubernetes MongoDB Operator, Ansible, ou MongoDB Atlas (cloud managé).
🌐 Réseau · ss

Pourquoi sudo ss -lntp renvoie plus d'informations (colonne Process) que ss -lntp sans sudo ?

La colonne Process qui affiche le nom et PID du processus propriétaire du socket nécessite l'accès aux informations des autres processus, ce qui requiert les droits root.

Sans sudo, ss peut voir les sockets mais pas identifier quel processus les a ouverts (sauf si le socket appartient à l'utilisateur courant). Avec sudo, on obtient la colonne users:(("mongod",pid=1130,fd=9)).
🔀 Sharding · Collection

À quel moment la collection est-elle créée ? Avantages/inconvénients de la création lazy ?

La collection est créée lors du premier insert (lazy creation). Dans ce TP, sh.shardCollection() crée la collection vide et la marque comme shardée avant tout insert.

Avantages : flexibilité du schéma, pas de configuration a priori, moins de boilerplate.

Inconvénients : si on insère AVANT de sharding une collection, tous les documents iront sur le shard primary de la base. Il faudra ensuite activer le sharding et attendre le rééquilibrage. C'est pourquoi on shard la collection AVANT d'importer les données.
🤖 Déploiement · Alternatives

Comment aurait-on pu installer ce cluster différemment ?

  • Docker Compose : déploiement rapide pour le développement (images officielles mongo)
  • Kubernetes + MongoDB Community Operator : déploiement cloud-native avec StatefulSets, gestion automatique des replica sets
  • Ansible : rôle community.mongodb pour automatiser l'installation sur de vraies VMs
  • MongoDB Atlas : service cloud managé — zero infrastructure, backup intégré, scaling automatique
  • Terraform + scripts : infrastructure as code reproductible
L'installation manuelle est utile pour comprendre l'architecture et diagnostiquer des problèmes. En production, on automatise toujours.
🔁 Déploiement · Reboot

Pourquoi redémarrer les serveurs avant de commencer à s'en servir ?

Un reboot est un test de validation du déploiement :
  • Vérifie que tous les services s'activent bien via systemctl enable au démarrage
  • Vérifie que les dépendances réseau sont respectées (services démarrent dans le bon ordre)
  • Teste que la configuration (fstab, hosts, etc.) survit à un redémarrage
  • Simule un crash non planifié → on sait que le système se relèvera seul
C'est une étape indispensable avant de mettre en production.

# Sauvegarde (dump) d'une collection vers BSON

# Déployer shard1-r1 (service mongod --shardsvr)

# Dans le .service : ExecStart=...mongod --shardsvr --replSet shard1 --port 28111 --dbpath /srv/data/mongo-shard1-r1 --bind_ip mongo-shard1-r1 --logpath=...
user@nosql010:~$ sudo systemctl restart mongo-shard1-r1 ; sleep 2 ; sudo systemctl status mongo-shard1-r1 --no-pager

# Initialiser le replica set du shard (1 seul nœud)
user@nosql010:~$ mongosh mongo-shard1-r1:28111
test> rs.initiate()  # config automatique pour un seul nœud
shard1 [direct: primary] test> rs.status()

# Déployer le router (mongos, pas mongod !)
# Dans le .service : ExecStart=/usr/bin/mongos --port 28001 --bind_ip mongo-router1 --configdb rs0/mongo-config-r1:28101,mongo-config-r2:28102,mongo-config-r3:28103 --logpath ...
user@nosql010:~$ sudo systemctl restart mongo-router1 ; sleep 2 ; sudo systemctl status mongo-router1 --no-pager

# Ajouter les shards au cluster via le router
user@nosql010:~$ mongosh mongo-router1:28001

[direct: mongos] test> sh.addShard("shard2/mongo-shard2-r1:28121")

# Activer le démarrage auto (APRÈS validation)
user@nosql010:~$ sudo systemctl enable {mongo-config-r1,mongo-config-r2,mongo-config-r3,mongo-shard1-r1,mongo-shard2-r1,mongo-router1}
user@nosql010:~$ sudo shutdown -r 0  # reboot de validation

mongosh · Sharding

# Réduire la taille des chunks pour les tests (16Mo)
[direct: mongos] test> use config
[direct: mongos] config> db.settings.updateOne(
  { _id: "chunksize" },
  { $set: { _id: "chunksize", value: 16 } },
  { upsert: true }
)

# Activer le sharding sur la base
[direct: mongos] config> use exampleDB
[direct: mongos] exampleDB> sh.enableSharding("exampleDB")

# Sharding d'une collection avec hash sur _id (distribution uniforme)
[direct: mongos] exampleDB> sh.shardCollection("exampleDB.restaurants", { "_id": "hashed" })

# Voir l'état du cluster et la distribution des chunks
[direct: mongos] exampleDB> sh.status()
[direct: mongos] exampleDB> db.restaurants.getShardDistribution()

# Sharding a posteriori (collection déjà peuplée)
[direct: mongos] exampleDB> db.trades.ensureIndex({ _id: "hashed" })
[direct: mongos] exampleDB> sh.shardCollection("exampleDB.trades", { "_id": "hashed" })

# Import de données vers le cluster shardé

# Script Python minimal (test.py)
import pymongo, time

# Connexion avec authentification
client = pymongo.MongoClient(
    host="localhost", port=27017,
    username="example-user", password="password",
    authSource="user-data"
)
db = client["exampleDB"]

# Lister les collections
print(db.list_collection_names())

# find_one avec filtre
restaurants = db["restaurants"]
restaurant = restaurants.find_one({"borough": "Queens"})
print(restaurant.get("name"))

# Mesure de performance
start = time.perf_counter()
count = restaurants.count_documents({"address.zipcode": "11375"})
elapsed = time.perf_counter() - start
print(f"{count} docs trouvés en {elapsed * 1000:.4f} ms")

# Exécuter le script
user@nosql010:~$ python3 test.py
