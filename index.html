📌 Résumé des Cours

1. NoSQL : En Bref

Définition : Bases de données non relationnelles, optimisées pour la scalabilité horizontale, la flexibilité des schémas et la gestion de grands volumes de données non structurées.

Types de NoSQL

Type Exemples Cas d'usage Avantages Inconvénients
Clé-Valeur Redis, DynamoDB Cache, sessions, métadonnées Simplicité, performance Limité aux requêtes par clé
Documents MongoDB, CouchDB Applications web, catalogues Flexibilité, requêtes riches Moins adapté aux jointures complexes
Colonnes Cassandra, HBase Big Data, séries temporelles Scalabilité massive, écritures rapides Requêtes complexes limitées
Graphes Neo4j, Amazon Neptune Réseaux sociaux, recommandations Requêtes relationnelles puissantes Performance limitée pour les grands graphes

Théorème CAP

Un système distribué ne peut garantir simultanément que 2 des 3 propriétés :

  • Cohérence (tous les nœuds voient les mêmes données)
  • Availability (chaque requête reçoit une réponse)
  • Partition tolerance (le système fonctionne malgré des pannes réseau)

Exemple : MongoDB privilégie CP (cohérence + tolérance aux pannes), tandis que Cassandra privilégie AP (disponibilité + tolérance aux pannes).

BASE vs ACID

ACID (SQL) BASE (NoSQL)
Atomicité, Cohérence, Isolation, Durabilité Basically Available, Soft state, Eventual consistency
Transactions strictes, cohérence immédiate Disponibilité prioritaire, cohérence eventualle
Idéal pour les données structurées et critiques Idéal pour la scalabilité et la flexibilité

2. MongoDB : Résumé

Définition : Base de données orientée documents (JSON/BSON), avec scalabilité horizontale, réplication et sharding.

Concepts Clés

  • Document : Unité de stockage (ex: { "nom": "Alice", "âge": 30 }).
  • Collection : Ensemble de documents (équivalent d'une table en SQL).
  • Index : Améliore les performances des requêtes (ex: db.collection.createIndex({ "age": 1 })).
  • Replica Set : Groupe de serveurs pour la haute disponibilité (1 primaire + N secondaires).
  • Sharding : Partitionnement horizontal pour répartir les données sur plusieurs machines.

Opérations CRUD

Opération Commande MongoDB Équivalent SQL
Create db.collection.insertOne({...}) INSERT INTO table VALUES (...)
Read db.collection.find({ "age": { $gt: 25 } }) SELECT * FROM table WHERE age > 25
Update db.collection.updateOne({ "nom": "Alice" }, { $set: { "âge": 31 } }) UPDATE table SET age=31 WHERE name='Alice'
Delete db.collection.deleteOne({ "nom": "Alice" }) DELETE FROM table WHERE name='Alice'

Agrégations

Pipeline d'opérations pour traiter et analyser les données :

db.orders.aggregate([
  { $match: { status: "completed" } },  // Filtre
  { $group: { _id: "$client", total: { $sum: "$montant" } } },  // Groupe et somme
  { $sort: { total: -1 } }  // Tri décroissant
])

3. Stack ELK : Résumé

Définition : Suite d'outils pour la collecte, le traitement, le stockage et la visualisation de données (logs, métriques, etc.).

Composants

Outil Rôle Exemple d'usage
Elasticsearch Moteur de recherche/distribué pour stocker et indexer les données. Recherche full-text, analyse de logs.
Logstash Collecte, transforme et transporte les données (ETL). Parsing de logs Apache, enrichissement des données.
Kibana Visualisation et exploration des données via des dashboards. Création de graphiques, alertes, cartes géographiques.
Beats Agents légers pour collecter des données (logs, métriques, etc.). Filebeat pour les logs, Metricbeat pour les métriques système.

Architecture Typique

Filebeat (sur les serveurs) → Logstash (traitement) → Elasticsearch (stockage/indexation) → Kibana (visualisation).

Cas d'usage

  • Analyse de logs (Apache, Nginx, applications).
  • Monitoring des performances (métriques système, APM).
  • Recherche full-text (documents, produits, utilisateurs).
  • Sécurité (détection d'intrusions via les logs).

1. NoSQL : Fondamentaux

1.1 Pourquoi NoSQL ?

Les bases de données NoSQL ont émergé en réponse aux limites des bases relationnelles (SQL) pour :

  • Scalabilité : Difficulté à scalaire horizontalement avec SQL (nécessite des solutions coûteuses comme le sharding manuel).
  • Flexibilité : Schéma fixe en SQL, difficile à modifier pour des données non structurées (JSON, XML).
  • Performance : Les jointures complexes en SQL ralentissent les requêtes sur de grands volumes.
  • Big Data : Besoin de gérer des 4V : Volume, Vélocité, Variété, Véracité.

1.2 Les 4 Types de NoSQL

  1. Clé-Valeur :
    • Stockage sous forme de paires clé-valeur (ex: { "user123": "{'nom': 'Alice', 'âge': 30}" }).
    • Avantages : Simplicité, performance extrême pour les lectures/écritures.
    • Cas d'usage : Cache (Redis), sessions utilisateurs, métadonnées.
  2. Documents :
    • Stockage de documents JSON/BSON (ex: MongoDB).
    • Avantages : Flexibilité du schéma, requêtes riches, embarquement de données liées.
    • Cas d'usage : Applications web, catalogues produits, profils utilisateurs.
  3. Colonnes (Wide-Column) :
    • Stockage par colonnes plutôt que par lignes (ex: Cassandra, HBase).
    • Avantages : Scalabilité massive, écritures rapides, optimisé pour les grands volumes.
    • Cas d'usage : Big Data, séries temporelles, IoT.
  4. Graphes :
    • Stockage de nœuds et de relations (ex: Neo4j).
    • Avantages : Requêtes relationnelles complexes (ex: "trouver tous les amis d'amis").
    • Cas d'usage : Réseaux sociaux, recommandations, détection de fraudes.

1.3 Théorème CAP

Dans un système distribué, il est impossible de garantir simultanément les trois propriétés suivantes :

  • Cohérence (Consistency) : Tous les nœuds voient les mêmes données au même moment.
  • Disponibilité (Availability) : Chaque requête reçoit une réponse, même en cas de panne partielle.
  • Tolérance au partitionnement (Partition tolerance) : Le système continue de fonctionner malgré des pannes réseau.

Conséquence : Les bases NoSQL choisissent généralement 2 des 3 propriétés :

Type de Base Propriétés Garanties Exemple
CP (Cohérence + Partition) Priorité à la cohérence, même en cas de partition. MongoDB (par défaut), etcd
AP (Disponibilité + Partition) Priorité à la disponibilité, cohérence eventualle. Cassandra, DynamoDB
CA (Cohérence + Disponibilité) Possible seulement sans partition (rare en pratique). Bases SQL traditionnelles (non distribuées)

1.4 BASE vs ACID

ACID (SQL) BASE (NoSQL)
Atomicité : Une transaction est entièrement exécutée ou pas du tout.
Cohérence : Les données sont toujours valides après une transaction.
Isolation : Les transactions concurrentes ne s'influencent pas.
Durabilité : Une transaction validée persiste même après une panne. 		Basically Available : Le système garantit la disponibilité.
Soft state : L'état peut changer sans intervention.
Eventual consistency : Les données deviennent cohérentes après un certain temps.
Avantages : Cohérence forte, transactions fiables.
Inconvénients : Scalabilité limitée, schéma rigide. 		Avantages : Scalabilité horizontale, flexibilité, haute disponibilité.
Inconvénients : Cohérence eventualle, complexité applicative.

1.5 Quand utiliser NoSQL ?

Cas d'usage idéaux pour NoSQL :
  • Données non structurées : JSON, XML, logs, médias.
  • Scalabilité massive : Millions d'utilisateurs ou de requêtes (ex: réseaux sociaux).
  • Flexibilité du schéma : Besoin d'ajouter/modifier des champs fréquemment.
  • Haute disponibilité : Tolérance aux pannes réseau ou matérielles.
  • Performance en écriture : Ex: IoT, métriques en temps réel.
Quand éviter NoSQL ?
  • Données fortement relationnelles (ex: comptabilité, ERP).
  • Besoin de transactions ACID strictes (ex: banques).
  • Requêtes complexes avec jointures.

2. MongoDB

2.1 Introduction et Concepts

MongoDB est une base de données NoSQL orientée documents, utilisant le format BSON (JSON binaire). Elle est conçue pour la flexibilité, la scalabilité et les performances.

2.1.1 Modèle de Données

  • Document : Unité de base (ex: { "_id": 1, "nom": "Alice", "âge": 30, "adresse": { "ville": "Paris" } }).
  • Collection : Ensemble de documents (équivalent d'une table en SQL).
  • Base de données : Conteneur pour les collections.

Exemple de document embarqué (évite les jointures) : 

{
  "_id": ObjectId("507f1f77bcf86cd799439011"),
  "nom": "Alice",
  "âge": 30,
  "adresse": {
    "rue": "123 Rue de Paris",
    "ville": "Paris",
    "code_postal": "75000"
  },
  "commandes": [
    { "id": "cmd1", "montant": 99.99, "date": ISODate("2023-10-01T00:00:00Z") },
    { "id": "cmd2", "montant": 49.99, "date": ISODate("2023-10-15T00:00:00Z") }
  ]
}

2.1.2 Opérations CRUD

Opération Méthode MongoDB Exemple Équivalent SQL
Create insertOne(), insertMany() db.clients.insertOne({ nom: "Bob", âge: 25 }) INSERT INTO clients VALUES (...)
Read find(), findOne() db.clients.find({ âge: { $gt: 20 } }) SELECT * FROM clients WHERE age > 20
Update updateOne(), updateMany() db.clients.updateOne({ nom: "Bob" }, { $set: { âge: 26 } }) UPDATE clients SET age=26 WHERE name='Bob'
Delete deleteOne(), deleteMany() db.clients.deleteOne({ nom: "Bob" }) DELETE FROM clients WHERE name='Bob'

2.1.3 Requêtes Avancées

Opérateurs de comparaison : $eq, $ne, $gt, $lt, $in, $nin.

Opérateurs logiques : $and, $or, $not.

Exemple : Trouver les clients de Paris âgés de plus de 25 ans :

db.clients.find({
  "adresse.ville": "Paris",
  "âge": { $gt: 25 }
})

2.1.4 Agrégations

Les agrégations permettent de traiter les données via un pipeline (séquence d'étapes).

Étapes courantes :

  • $match : Filtre les documents.
  • $group : Regroupe les documents par clé.
  • $sort : Trie les résultats.
  • $project : Sélectionne les champs à retourner.
  • $limit : Limite le nombre de résultats.

Exemple : Calculer le montant total des commandes par client :

db.commandes.aggregate([
  { $match: { "date": { $gte: ISODate("2023-01-01") } } },
  { $group: {
    _id: "$client_id",
    total: { $sum: "$montant" },
    count: { $sum: 1 }
  }},
  { $sort: { total: -1 } }
])

2.1.5 Index

Les index améliorent les performances des requêtes en permettant un accès rapide aux documents.

  • Création : db.collection.createIndex({ "champ": 1 }) (1 = croissant, -1 = décroissant).
  • Types :
    • Index simple (un champ).
    • Index composé (plusieurs champs).
    • Index textuel (recherche full-text).
    • Index géospatial (coordonnées GPS).
  • Exemple : Créer un index sur le champ âge : 
    db.clients.createIndex({ "âge": 1 })

Attention : Les index accélèrent les lectures mais ralentissent les écritures (chaque index doit être mis à jour).

2.1.6 Réplication (Replica Sets)

Un replica set est un groupe de serveurs MongoDB qui maintiennent les mêmes données pour assurer la haute disponibilité.

  • Nœud primaire : Reçoit toutes les écritures.
  • Nœuds secondaires : Répliquent les données du primaire (lectures possibles).
  • Arbitre (optionnel) : Participe aux élections sans stocker de données.

Avantages :

  • Tolérance aux pannes (si le primaire tombe, un secondaire est élu).
  • Lectures scalables (répartition sur les secondaires).
  • Sauvegardes sans impact sur le primaire.

2.1.7 Sharding

Le sharding permet de répartir les données d'une collection sur plusieurs machines (shards) pour améliorer les performances et la capacité de stockage.

  • Shard : Stocke une partie des données.
  • Config Server : Stocke les métadonnées du cluster.
  • Mongos : Routeur qui achemine les requêtes vers les shards.

Clé de sharding : Champ utilisé pour répartir les données (ex: _id, client_id).

Stratégies de sharding :

  • Hachage : Bonne distribution, mais moins efficace pour les requêtes par plage.
  • Plage : Efficace pour les requêtes par plage, mais peut déséquilibrer les shards.
  • Clé composée : Combine plusieurs champs pour optimiser la distribution.

2.2 TP MongoDB avec Réponses

2.2.1 Installation Standalone

Q : Pourquoi installer MongoDB en mode standalone ?
R :
  • Environnements de développement/test (simplicité).
  • Applications avec faible volume de données.
  • Prototypage ou démonstrations (pas besoin de haute disponibilité).

2.2.2 Sécurité et Authentification

Q : Pourquoi le message "Access control is not enabled" apparaît-il après avoir configuré la sécurité ?
R :

La configuration de la sécurité (dans /etc/mongod.conf) nécessite un redémarrage du service pour prendre effet. Tant que MongoDB n'est pas redémarré, il fonctionne sans contrôle d'accès.

Q : Quels risques si on active l'authentification sans créer d'utilisateur administrateur ?
R :

On risque de perdre l'accès à la base de données, car aucun utilisateur ne pourra se connecter pour créer des comptes ou modifier la configuration.

Solution : Toujours créer un utilisateur admin avant d'activer l'authentification : 

use admin
db.createUser({
  user: "admin",
  pwd: "motdepasse",
  roles: ["root"]
})

2.2.3 Manipulation des Données

Q : Pourquoi la base "user-data" n'est-elle pas visible avant d'y créer une collection ?
R :

MongoDB utilise une création paresseuse : une base ou une collection n'apparaît que lorsqu'elle contient des données (ou qu'une collection y est explicitement créée).

2.2.4 Import/Export de Données

Q : À quoi sert la commande sha256sum sur le fichier primer-dataset.json ?
R :

Elle calcule une empreinte cryptographique (hash) du fichier pour :

  • Vérifier son intégrité (pas de corruption lors du téléchargement).
  • Comparer avec une somme de contrôle attendue (ex: fournie par le site source).

Q : Que fait le "one-liner" pushd /tmp ; wget ... ; popd ?
R :

C'est une séquence de commandes en une ligne :

  • pushd /tmp : Change de répertoire vers /tmp et sauvegarde l'ancien.
  • wget ... : Télécharge un fichier depuis GitHub.
  • popd : Retourne au répertoire précédent.

2.2.5 Sharded Cluster

Q : Pourquoi utiliser des adresses en 127.0.3.x dans ce lab ?
R :

Ces adresses font partie de la plage localhost (127.0.0.0/8) et permettent de :

  • Simuler plusieurs interfaces réseau sur une seule machine.
  • Éviter les conflits avec d'autres adresses IP réelles.
  • Tester un cluster sans besoin de plusieurs machines physiques.

Q : Pourquoi nommer les dossiers de données différemment pour chaque service (config server, shard, etc.) ?
R :

Pour :

  • Éviter les conflits entre les instances.
  • Isoler les données de chaque service (facilite la maintenance et le débogage).
  • Permettre une gestion indépendante (ex: sauvegardes ciblées).

2.2.6 Index et Performances

Q : Comment un index améliore-t-il les performances des requêtes ?
R :

Sans index, MongoDB doit effectuer un scanning complet de la collection (COLLSCAN), ce qui est lent pour les grandes collections.

Avec un index, MongoDB utilise un scan d'index (IXSCAN), qui permet un accès direct aux documents pertinents, comme un index dans un livre.

Exemple : Sans index sur age, la requête db.clients.find({ age: { $gt: 25 } }) scanne tous les documents. Avec un index, elle utilise l'arbre B de l'index pour trouver rapidement les documents.

3. Stack ELK

3.1 Introduction et Composants

La stack ELK est une solution open-source pour la collecte, le traitement, le stockage et la visualisation de données, notamment des logs.

3.1.1 Elasticsearch

Moteur de recherche et d'analyse distribué, basé sur Apache Lucene. Il stocke les données sous forme de documents JSON dans des index.

  • Shards : Partitionnement horizontal des index.
  • Répliques : Copies des shards pour la redondance.
  • Requêtes : Support des requêtes full-text, des agrégations et des filtres.

3.1.2 Logstash

Outil de traitement de données (ETL) avec un pipeline en 3 étapes :

  1. Input : Source des données (fichiers, bases de données, APIs).
  2. Filter : Transformation des données (parsing, enrichissement).
  3. Output : Destination (Elasticsearch, fichiers, etc.).

Exemple de configuration :

input {
  file {
    path => "/var/log/apache2/access.log"
    start_position => "beginning"
  }
}
filter {
  grok {
    match => { "message" => "%{COMBINEDAPACHELOG}" }
  }
  date {
    match => [ "timestamp", "dd/MMM/yyyy:HH:mm:ss Z" ]
  }
}
output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "apache-logs-%{+YYYY.MM.dd}"
  }
}

3.1.3 Kibana

Interface web pour visualiser et explorer les données stockées dans Elasticsearch.

  • Discover : Exploration interactive des données.
  • Visualize : Création de graphiques (histogrammes, cartes, etc.).
  • Dashboard : Tableaux de bord personnalisés.
  • Alerting : Configuration d'alertes basées sur des seuils.

3.1.4 Beats

Agents légers pour collecter des données et les envoyer à Logstash ou Elasticsearch.

Beat Type de Données Exemple d'usage
Filebeat Fichiers de logs Logs Apache, Nginx, applications
Metricbeat Métriques système CPU, mémoire, disque, réseau
Packetbeat Trafic réseau Analyse des paquets HTTP, DNS
Heartbeat Disponibilité des services Monitoring des uptimes

3.2 TP ELK avec Réponses

3.2.1 Installation d'Elasticsearch

Q : Que signifient -Xms512m et -Xmx512m dans jvm.options ?
R :

Ces paramètres définissent la taille de la mémoire heap allouée à la JVM (Java Virtual Machine) :

  • -Xms512m : Taille initiale de la heap (512 Mo).
  • -Xmx512m : Taille maximale de la heap (512 Mo).

En fixant -Xms et -Xmx à la même valeur, on évite à la JVM de redimensionner dynamiquement la heap, ce qui améliore les performances.

Q : Pourquoi ne pas activer le démarrage automatique d'Elasticsearch dès l'installation ?
R :

Pour :

  • Vérifier que le service démarre correctement en mode manuel.
  • S'assurer que la configuration est valide avant de l'automatiser.
  • Éviter que des erreurs de configuration ne bloquent le démarrage du système.

3.2.2 Installation de Kibana

Q : Pourquoi utiliser une redirection de port SSH plutôt qu'un navigateur sur le serveur ?
R :

Pour :

  • Accéder à Kibana depuis sa machine locale sans exposer le serveur.
  • Éviter d'installer un navigateur sur le serveur (sécurité et ressources).
  • Centraliser la gestion des accès (via SSH).

Exemple de commande : 

ssh -L 127.0.0.1:15601:127.0.0.1:5601 -l user -p 22 serveur_ip

Cette commande redirige le port local 15601 vers le port 5601 (Kibana) sur le serveur.

3.2.3 Configuration de Logstash

Q : Quel est le nom générique des outils comme Logstash ?
R :

Ces outils sont appelés des ETL (Extract, Transform, Load) ou des pipelines de traitement de données.

Q : À quoi sert la commande vmstat -S M 60 avec un timestamp ?
R :

Elle surveille les performances système (CPU, mémoire, E/S) toutes les 60 secondes, avec un timestamp ajouté pour :

  • Corréler les métriques avec les événements (ex: exécution de Logstash).
  • Analyser l'impact des traitements sur les ressources.

3.2.4 Analyse de Logs

Q : Pourquoi s'inquiéter de la synchronisation de l'heure sur les serveurs ?
R :

Pour garantir :

  • La cohérence des timestamps dans les logs.
  • Un séquençage correct des événements (ex: logs dans l'ordre chronologique).
  • Le bon fonctionnement des certificats TLS et des mécanismes de sécurité.

Q : Pourquoi utiliser /etc/hosts plutôt qu'un DNS dans ce lab ?
R :

Pour :

  • Simuler un DNS local sans serveur DNS.
  • Définir des alias (ex: mongo-config-r1) facilement.
  • Faciliter les tests en environnement isolé.

En production, on utiliserait un DNS interne (Bind, PowerDNS) ou un système de gestion de configuration (Ansible).

3.2.5 Visualisation avec Kibana

Q : Pourquoi la visualisation par défaut est-elle sur les 15 dernières minutes ?
R :

Pour :

  • Se concentrer sur les données récentes (pertinentes pour le monitoring).
  • Éviter de surcharger l'interface avec trop de données.

Modifier la période :

  1. Dans Kibana, aller dans Discover ou un dashboard.
  2. Cliquer sur le sélecteur de temps en haut à droite.
  3. Choisir une période prédéfinie ou personnalisée.

Q : Quelles visualisations sont utiles pour analyser les logs Apache ?
R :

Exemples de visualisations pertinentes :

  • Histogramme des codes HTTP : Détecter les erreurs (404, 500).
  • Top URLs : Identifier les pages les plus visitées.
  • Répartition par IP client : Détecter des attaques (DDoS).
  • Temps de réponse par URL : Trouver les pages lentes.
  • User-Agents : Connaître les navigateurs/robots.
  • Carte géographique : Visualiser la provenance des requêtes.

3.2.6 Import de Données

Q : À quoi sert l'option --dryRun dans mongorestore ?
R :

Elle permet de :

  • Simuler une restauration sans importer les données.
  • Vérifier que la commande est valide et que les données sont lisibles.
  • Estimer le temps et les ressources nécessaires.

Q : Pourquoi utiliser time pour mesurer l'import des données ?
R :

Pour :

  • Évaluer les performances de l'import.
  • Comparer différents scénarios (ex: avec/sans index).
  • Identifier des goulots d'étranglement (ex: temps d'indexation).

Sortie de time :

  • real : Temps total écoulé (inclut l'attente des E/S).
  • user : Temps CPU en mode utilisateur.
  • sys : Temps CPU en mode noyau.

Q : Pourquoi ne pas faire le tri côté client pour de grands volumes de données ?
R :

Car :

  • Le client devrait télécharger tous les documents avant de trier (coûteux en bande passante).
  • Les performances seraient dégradées par le transfert et le traitement local.
  • Elasticsearch est optimisé pour le tri côté serveur (index, algorithmes efficaces).

Q : Qu'est-ce que la pagination dans Elasticsearch ?
R :

La pagination divise les résultats en pages pour :

  • Éviter de surcharger le client avec trop de données.
  • Améliorer les performances et la réactivité.

Méthodes :

  • from et size :
    • from : Indice du premier document.
    • size : Nombre de documents par page.
    • Limite : from + size ≤ 10 000 par défaut.
  • search_after :
    • Utilise le dernier document de la page précédente pour obtenir la suivante.
    • Plus efficace pour les grands jeux de données.

4. Réponses aux Questions des TP

4.1 Questions Générales sur NoSQL

Q : Quels sont les avantages et inconvénients de BASE par rapport à ACID ?
R :

Avantages de BASE :

  • Scalabilité : Adapté aux systèmes distribués et grands volumes.
  • Disponibilité : Priorité à la disponibilité même en cas de partition.
  • Performance : Moins de surcharge liée aux transactions.

Inconvénients de BASE :

  • Cohérence eventualle : Les données peuvent être temporairement incohérentes.
  • Complexité applicative : L'application doit gérer les incohérences.
  • Pas de rollback : Difficile d'annuler une opération.

Q : Quand utiliser NoSQL plutôt que SQL ?
R :

Préférer NoSQL pour :

  • Données non structurées (JSON, logs).
  • Besoin de scalabilité horizontale (millions d'utilisateurs).
  • Flexibilité du schéma (ajout/modification fréquente de champs).
  • Haute disponibilité (tolérance aux pannes).

Préférer SQL pour :

  • Données fortement relationnelles (comptabilité, ERP).
  • Besoin de transactions ACID (banques).
  • Requêtes complexes avec jointures.

4.2 Questions sur MongoDB

Q : Pourquoi MongoDB utilise-t-il des documents JSON plutôt que des tables relationnelles ?
R :

Pour :

  • Flexibilité : Pas de schéma fixe, champs variables entre documents.
  • Correspondance naturelle avec les objets applicatifs (JSON).
  • Performance : Moins de jointures (données liées embarquées).
  • Scalabilité : Documents faciles à répartir (sharding).

Q : Comment choisir une clé de sharding dans MongoDB ?
R :

Critères :

  • Cardinalité élevée : Beaucoup de valeurs distinctes pour éviter les "hotspots".
  • Distribution uniforme : Répartition équilibrée des requêtes.
  • Stabilité : Clé qui ne change pas fréquemment.
  • Fréquence d'usage : Clé souvent utilisée dans les requêtes.

Stratégies :

  • Hachage : Bonne distribution, moins efficace pour les requêtes par plage.
  • Plage : Efficace pour les plages, mais peut déséquilibrer les shards.
  • Clé composée : Combine plusieurs champs.

4.3 Questions sur la Stack ELK

Q : Quelles sont les différences entre Logstash et Beats ?
R :
Critère Logstash Beats
Type Outil ETL lourd (JVM) Agents légers (Go)
Fonctionnalités Transformations complexes (filtrage, parsing) Collecte simple (logs, métriques)
Ressources Gourmand en mémoire/CPU Léger, adapté aux environnements embarqués
Cas d'usage Pipelines avancés (agrégation de sources multiples) Collecte distribuée (ex: Filebeat sur chaque serveur)

Complémentarité : Les Beats collectent les données et les envoient à Logstash pour traitement, ou directement à Elasticsearch.

Q : Comment optimiser Elasticsearch pour les performances ?
R :

Stratégies :

  • Index et mappings :
    • Définir des mappings explicites (éviter le mapping dynamique).
    • Utiliser des types de champs adaptés (ex: keyword pour les champs triables).
  • Sharding :
    • Choisir un nombre de shards adapté à la taille des données.
    • Éviter les shards trop petits ou trop grands.
  • Réplication :
    • Ajuster le nombre de répliques selon les besoins en disponibilité.
  • Matériel :
    • Allouer suffisamment de RAM (heap JVM + cache système).
    • Utiliser des disques SSD pour les performances d'E/S.
  • Requêtes :
    • Privilégier les filtres (filter) aux requêtes (query) pour les critères booléens.
    • Limiter le nombre de résultats avec size.

5. Conclusion et Ressources

Ce cours a couvert les fondamentaux de NoSQL, MongoDB et la stack ELK, ainsi que leurs applications pratiques via des TP. Voici les points clés à retenir :

  • NoSQL : Alternative flexible et scalable aux bases relationnelles, idéale pour les données non structurées et les grands volumes.
  • MongoDB : Base de données orientée documents, avec réplication, sharding et requêtes riches.
  • Stack ELK : Solution complète pour la collecte, le traitement et la visualisation de données (logs, métriques, etc.).
