1. Introduction à NoSQL

1.1. Définition et Concepts Clés

NoSQL (Not Only SQL) désigne une famille de systèmes de gestion de bases de données qui ne reposent pas sur le modèle relationnel traditionnel. Ces bases de données sont conçues pour gérer des volumes massifs de données non structurées ou semi-structurées, avec une grande flexibilité et une scalabilité horizontale.

Caractéristiques principales de NoSQL :
  • Flexibilité du schéma : Pas de schéma fixe, les données peuvent être ajoutées ou modifiées sans contrainte.
  • Scalabilité horizontale : Possibilité d'ajouter des nœuds pour gérer des charges accrues.
  • Haute disponibilité : Conçu pour fonctionner même en cas de panne de certains nœuds.
  • Performance : Optimisé pour des lectures/écritures rapides sur de grands volumes de données.

1.2. ACID vs BASE

ACID (Bases de données relationnelles) BASE (Bases de données NoSQL)
Atomicité : Une transaction est soit entièrement exécutée, soit pas du tout. Basically Available : Le système garantit la disponibilité des données, même en cas de panne partielle.
Cohérence : Les données sont toujours dans un état valide après une transaction. Soft state : L'état du système peut changer avec le temps, même sans intervention.
Isolation : Les transactions concurrentes ne s'influencent pas mutuellement. Eventual consistency : Les données deviennent cohérentes après un certain temps.
Durabilité : Une fois validée, une transaction persiste même en cas de panne. La durabilité est souvent sacrifiée pour la performance et la disponibilité.

1.3. Théorème CAP

Le théorème CAP (Consistency, Availability, Partition tolerance) stipule qu'un système distribué ne peut garantir simultanément plus de deux des trois propriétés suivantes :

  • Cohérence (Consistency) : Tous les nœuds voient les mêmes données au même moment.
  • Disponibilité (Availability) : Chaque requête reçoit une réponse, même en cas de panne.
  • Tolérance au partitionnement (Partition tolerance) : Le système continue de fonctionner même si des nœuds sont isolés.

En pratique, les systèmes NoSQL privilégient souvent la disponibilité et la tolérance au partitionnement au détriment de la cohérence immédiate.

1.4. Les 4 V du Big Data

  • Volume : Quantité massive de données générées.
  • Vélocité : Vitesse à laquelle les données sont générées et traitées.
  • Variété : Divers formats de données (texte, images, vidéos, etc.).
  • Véracité : Qualité et fiabilité des données.

2. MongoDB

2.1. Introduction à MongoDB

MongoDB est une base de données NoSQL orientée documents. Elle stocke les données sous forme de documents JSON (BSON en interne), ce qui permet une grande flexibilité et une scalabilité horizontale.

Caractéristiques principales :
  • Stockage sous forme de documents JSON.
  • Requêtes riches avec un langage de requête JSON.
  • Support des index, agrégations, et requêtes géospatiales.
  • Réplication et sharding pour la scalabilité.
  • Support des transactions ACID (depuis la version 4.0).

2.1.1. Modèle de Données

MongoDB utilise un modèle de données basé sur des collections (équivalent des tables en SQL) et des documents (équivalent des lignes en SQL). Chaque document est un objet JSON avec des paires clé-valeur. 

{
  "_id": ObjectId("507f1f77bcf86cd799439011"),
  "name": "John Doe",
  "age": 30,
  "address": {
    "street": "123 Main St",
    "city": "New York",
    "zip": "10001"
  },
  "hobbies": ["reading", "hiking", "coding"]
}

2.1.2. Opérations CRUD

MongoDB supporte les opérations CRUD (Create, Read, Update, Delete) via des méthodes simples :

  • Create : db.collection.insertOne() ou insertMany()
  • Read : db.collection.find() ou findOne()
  • Update : db.collection.updateOne() ou updateMany()
  • Delete : db.collection.deleteOne() ou deleteMany()

2.1.3. Agrégations

MongoDB permet d'effectuer des agrégations complexes via un pipeline d'agrégation, qui est une séquence d'opérations appliquées aux documents. 

db.orders.aggregate([
  { $match: { status: "completed" } },
  { $group: { _id: "$customerId", total: { $sum: "$amount" } } },
  { $sort: { total: -1 } }
])

2.1.4. Index

Les index améliorent les performances des requêtes en permettant un accès rapide aux documents. MongoDB supporte plusieurs types d'index :

  • Index sur un seul champ : db.collection.createIndex({ field: 1 })
  • Index composé : db.collection.createIndex({ field1: 1, field2: -1 })
  • Index textuel : db.collection.createIndex({ content: "text" })
  • Index géospatial : db.collection.createIndex({ location: "2dsphere" })

2.1.5. Réplication et Sharding

Réplication : MongoDB utilise des replica sets pour assurer la haute disponibilité. Un replica set est un groupe de serveurs MongoDB qui maintiennent les mêmes données, offrant une redondance et une tolérance aux pannes.

Sharding : Le sharding permet de répartir les données sur plusieurs machines (shards) pour améliorer les performances et la scalabilité. Un config server stocke les métadonnées du cluster, et un mongos (routeur) achemine les requêtes vers les shards appropriés.

2.2. TP MongoDB

2.2.1. Installation de MongoDB Standalone

Question : Pour quels cas d'usage voudrait-on installer un MongoDB standalone ?
Réponse :

Un MongoDB standalone est adapté pour :

  • Les environnements de développement ou de test où la haute disponibilité n'est pas critique.
  • Les petites applications avec un faible volume de données et de requêtes.
  • Les démonstrations ou prototypes où la simplicité prime sur la résilience.

2.2.2. Configuration de MongoDB

Question : Pourquoi va-t-on avoir ce message « Access control is not enabled for the database », alors qu'on vient de configurer la sécurité ?
Réponse :

Ce message apparaît car la configuration de la sécurité (activation de l'authentification) nécessite un redémarrage du service MongoDB pour prendre effet. Tant que le service n'est pas redémarré, MongoDB fonctionne sans contrôle d'accès.

Question : Qu'est-ce qui pourrait mal se passer dans cette procédure d'activation de l'authentification ? Que pourrait-on faire dans un ordre différent ?
Réponse :

Plusieurs problèmes peuvent survenir :

  • Oublier de créer un utilisateur administrateur avant d'activer l'authentification, ce qui peut verrouiller l'accès à la base.
  • Ne pas redémarrer le service après avoir modifié la configuration, ce qui laisse le système non sécurisé.
  • Erreurs de syntaxe dans le fichier de configuration (/etc/mongod.conf).

Pour éviter ces problèmes, il est recommandé de :

  1. Créer un utilisateur administrateur avant d'activer l'authentification.
  2. Vérifier la syntaxe du fichier de configuration.
  3. Redémarrer le service après toute modification.

2.2.3. Manipulation des Données

Question : Pourquoi ne voit-on pas la base « user-data » avant d'y créer une collection ?
Réponse :

MongoDB utilise un mécanisme de création paresseuse (lazy creation). Une base de données ou une collection n'apparaît pas tant qu'elle ne contient pas de données. Ainsi, la base « user-data » n'est visible qu'après la création d'une collection ou l'insertion de données.

2.2.4. Import de Données

Question : Que fait le « one-liner » pushd /tmp ; wget https://raw.githubusercontent.com/mongodb/docs-assets/primer-dataset/primer-dataset.json ; popd /tmp ~ ? Qu'est-ce qu'un « one-liner » ?
Réponse :

Ce « one-liner » est une commande unique qui effectue plusieurs actions :

  1. pushd /tmp : Change le répertoire courant pour /tmp et sauvegarde le répertoire précédent dans une pile.
  2. wget https://raw.githubusercontent.com/... : Télécharge le fichier primer-dataset.json depuis GitHub.
  3. popd : Revient au répertoire précédent (avant le pushd).

Un « one-liner » est une commande ou un script condensé en une seule ligne, souvent utilisé pour effectuer une série d'opérations rapidement.

Question : Que fait la commande sha256sum /tmp/primer-dataset.json ? À quoi cela sert-il ?
Réponse :

La commande sha256sum calcule une empreinte cryptographique (hash) du fichier primer-dataset.json en utilisant l'algorithme SHA-256. Cela permet de :

  • Vérifier l'intégrité du fichier (s'assurer qu'il n'a pas été modifié).
  • Comparer le fichier téléchargé avec une somme de contrôle attendue pour détecter toute corruption ou altération.

2.2.5. Sharded Cluster

Question : Pourquoi reprendre la nomenclature et l'organisation des ressources (FQDN:Port) d'un cluster complexe pour créer un cluster simple ?
Réponse :

Même pour un cluster simple, utiliser une nomenclature cohérente avec celle d'un cluster complexe permet de :

  • Faciliter la compréhension et la maintenance du cluster.
  • Préparer une évolution future vers un cluster plus complexe sans avoir à renommer les ressources.
  • Standardiser les pratiques et les scripts de déploiement.
  • Simuler un environnement de production même en développement ou en test.
Question : Qu'est-ce qu'un FQDN ?
Réponse :

Un FQDN (Fully Qualified Domain Name) est le nom complet d'un hôte sur un réseau, incluant le nom de l'hôte et le nom de domaine. Par exemple, mongo-config-r1.lab.local est un FQDN où :

  • mongo-config-r1 est le nom de l'hôte.
  • lab.local est le nom de domaine.
Question : Que sont ces adresses en 127.0.3.x ? Pourquoi les utiliser dans ce lab ?
Réponse :

Les adresses 127.0.3.x font partie de la plage 127.0.0.0/8, réservée pour la boucle locale (localhost). Elles sont utilisées dans ce lab pour :

  • Simuler plusieurs interfaces réseau sur une seule machine.
  • Éviter les conflits avec d'autres adresses IP réelles.
  • Tester des configurations de cluster sans avoir besoin de plusieurs machines physiques.

2.2.6. Configuration des Services

Question : Pourquoi nommer le dossier /data/mongo... différemment sur chaque serveur ?
Réponse :

Chaque instance MongoDB (config server, shard, router) doit avoir son propre dossier de données pour :

  • Éviter les conflits entre les différentes instances.
  • Permettre une gestion indépendante des données de chaque service.
  • Faciliter le débogage et la maintenance en isolant les données.
Question : Pourquoi passer les services en démarrage automatique ?
Réponse :

Activer le démarrage automatique des services (via systemctl enable) permet de :

  • Assurer que les services redémarrent automatiquement après un redémarrage du serveur.
  • Éviter de devoir lancer manuellement chaque service après une panne ou une maintenance.
  • Garantir la disponibilité continue des services en production.
Question : Pourquoi attendre la toute fin de sa création avant de passer le service en démarrage automatique ?
Réponse :

Il est préférable d'attendre la fin de la création et de la configuration pour activer le démarrage automatique afin de :

  • Éviter que le service ne démarre avec une configuration incomplète.
  • Pouvoir tester manuellement le service avant de l'automatiser.
  • Corriger d'éventuelles erreurs de configuration avant qu'elles n'affectent le démarrage automatique.

2.2.7. Index et Performances

Question : Commentez la création de l'index, et son effet sur le plan d'exécution.
Réponse :

La création d'un index sur un champ (par exemple db.restaurants.createIndex({ "address.zipcode": 1 })) permet d'améliorer significativement les performances des requêtes qui filtrent ou trient sur ce champ. Avant la création de l'index, MongoDB doit effectuer un scanning complet de la collection (COLLSCAN), ce qui est coûteux en temps et en ressources.

Après la création de l'index, MongoDB utilise un scan d'index (IXSCAN), qui est beaucoup plus rapide car il permet un accès direct aux documents pertinents sans parcourir toute la collection.

Cependant, les index ont un coût :

  • Ils consomment de l'espace disque supplémentaire.
  • Ils ralentissent les écritures (insertions, mises à jour, suppressions) car chaque index doit être mis à jour.

3. Stack ELK

3.1. Introduction à la Stack ELK

La stack ELK (Elasticsearch, Logstash, Kibana) est une solution populaire pour la collecte, le traitement, l'indexation et la visualisation de données, notamment des logs. Elle est souvent utilisée pour l'analyse de logs, la recherche full-text, et la visualisation de données en temps réel.

Composants principaux :
  • Elasticsearch : Moteur de recherche et de stockage distribué.
  • Logstash : Outil de collecte, de transformation et de transport de données.
  • Kibana : Interface de visualisation et d'exploration des données.
  • Beats/Elastic Agent : Agents légers pour la collecte de données (logs, métriques, etc.).

3.1.1. Elasticsearch

Elasticsearch est un moteur de recherche distribué basé sur Apache Lucene. Il est conçu pour le stockage, la recherche et l'analyse de grandes quantités de données en temps quasi réel.

  • Stockage sous forme de documents JSON dans des index.
  • Support des requêtes full-text, des agrégations, et des requêtes géospatiales.
  • Architecture distribuée avec sharding et réplication.

3.1.2. Logstash

Logstash est un outil de traitement de données qui permet de collecter des données depuis diverses sources, de les transformer, et de les envoyer vers une destination (souvent Elasticsearch).

  • Pipeline en trois étapes : Input → Filter → Output.
  • Support de nombreux plugins pour la collecte (fichiers, bases de données, APIs, etc.) et la transformation (parsing, enrichissement, normalisation).

3.1.3. Kibana

Kibana est une interface web pour visualiser et explorer les données stockées dans Elasticsearch. Elle permet de créer des tableaux de bord (dashboards), des graphiques, et des requêtes interactives.

  • Exploration des données avec Discover.
  • Création de visualisations avec Visualize ou Lens.
  • Gestion des index et des configurations via Stack Management.

3.1.4. Beats et Elastic Agent

Les Beats (comme Filebeat, Metricbeat) et Elastic Agent sont des agents légers qui collectent des données (logs, métriques, etc.) et les envoient à Logstash ou directement à Elasticsearch.

  • Filebeat : Collecte et envoie des fichiers de logs.
  • Metricbeat : Collecte des métriques système.
  • Elastic Agent : Agent unifié pour déployer et gérer plusieurs Beats.

3.2. TP ELK

3.2.1. Installation d'Elasticsearch

Question : Que signifient les paramètres -Xms512m et -Xmx512m dans le fichier jvm.options ?
Réponse :

Ces paramètres définissent la taille de la mémoire heap allouée à la JVM (Java Virtual Machine) qui exécute Elasticsearch :

  • -Xms512m : Définit la taille initiale de la heap à 512 Mo.
  • -Xmx512m : Définit la taille maximale de la heap à 512 Mo.

En fixant ces deux valeurs à la même taille, on évite à la JVM de redimensionner dynamiquement la heap, ce qui peut améliorer les performances et la stabilité.

Question : Pourquoi n'a-t-on pas activé le démarrage automatique du service Elasticsearch dès l'installation ?
Réponse :

Il est préférable de ne pas activer immédiatement le démarrage automatique pour :

  • Vérifier que le service démarre correctement en mode manuel.
  • S'assurer que la configuration est correcte avant de l'automatiser.
  • Éviter que des erreurs de configuration ne bloquent le démarrage du système.

3.2.2. Installation de Kibana

Question : Pourquoi utiliser une redirection de port vers votre hôte plutôt que d'installer un navigateur sur votre serveur ?
Réponse :

Utiliser une redirection de port (via SSH) permet de :

  • Accéder à l'interface Kibana depuis votre machine locale sans exposer le serveur.
  • Éviter d'installer des logiciels inutiles (comme un navigateur) sur le serveur.
  • Centraliser la gestion des accès et des pare-feux.
Question : Comment interpréter la commande ssh -L 127.0.0.1:15601:127.0.0.1:5601 -l user -p <port> -v <adresse_ip> ?
Réponse :

Cette commande SSH crée un tunnel SSH qui redirige le port local 15601 vers le port 5601 (Kibana) sur le serveur distant. Voici la décomposition :

  • -L 127.0.0.1:15601:127.0.0.1:5601 : Redirige le port local 15601 vers 127.0.0.1:5601 sur le serveur.
  • -l user : Spécifie l'utilisateur pour la connexion SSH.
  • -p <port> : Spécifie le port SSH du serveur (par défaut 22).
  • -v : Active le mode verbeux pour le débogage.
  • <adresse_ip> : Adresse IP du serveur distant.

Ainsi, en accédant à http://127.0.0.1:15601 sur votre machine locale, vous accédez en réalité à Kibana sur le serveur distant.

3.2.3. Installation de Logstash

Question : Quel nom générique donne-t-on aux produits qui ont les fonctionnalités de Logstash (collecte, transformation, transport) ?
Réponse :

Les outils comme Logstash sont généralement appelés des ETL (Extract, Transform, Load) ou des pipelines de traitement de données. Ils sont utilisés pour :

  • Extraire (Extract) des données depuis diverses sources.
  • Transformer (Transform) les données (nettoyage, enrichissement, normalisation).
  • Charger (Load) les données vers une destination (base de données, fichier, etc.).
Question : Comment interpréter la commande sudo sed 's/#--max-old-space-size=4096/--max-old-space-size=1024/' -i /etc/kibana/node.options ?
Réponse :

Cette commande utilise sed pour modifier le fichier /etc/kibana/node.options :

  • s/#--max-old-space-size=4096/--max-old-space-size=1024/ : Remplace la ligne commentée #--max-old-space-size=4096 par --max-old-space-size=1024.
  • -i : Modifie le fichier en place (in-place).

Cela réduit la mémoire maximale allouée à Kibana de 4 Go à 1 Go, ce qui est suffisant pour un environnement de test ou de développement.

3.2.4. Configuration de Logstash

Question : À quoi sert la commande vmstat -S M 60 | (while read line ; do echo -e "$(date '+%Y-%m-%d %H:%M:%S %z')|$line"; done) ?
Réponse :

Cette commande permet de surveiller les performances système toutes les 60 secondes et d'ajouter un timestamp à chaque ligne de sortie :

  • vmstat -S M 60 : Affiche les statistiques système (CPU, mémoire, E/S) toutes les 60 secondes, avec des unités en mégaoctets (-S M).
  • while read line ; do echo -e "$(date '+%Y-%m-%d %H:%M:%S %z')|$line"; done : Pour chaque ligne de sortie de vmstat, ajoute un timestamp au format AAAA-MM-JJ HH:MM:SS fuseau_horaire.

Cela permet de corréler les performances système avec les événements (comme l'exécution de Logstash).

Question : Pourquoi s'inquiéter de mettre notre serveur à l'heure ?
Réponse :

La synchronisation de l'heure est cruciale pour :

  • Garantir la cohérence des timestamps dans les logs et les données.
  • Éviter les problèmes de séquençage des événements (par exemple, des logs dans le désordre).
  • Assurer le bon fonctionnement des mécanismes de sécurité (certificats TLS, tokens d'authentification).
  • Faciliter le débogage et l'analyse des logs.

3.2.5. Analyse de Logs avec Logstash

Question : Pourquoi utilise-t-on la résolution par le fichier /etc/hosts dans ce lab ? Comment ferait-on dans le SI d'une organisation ?
Réponse :

Dans ce lab, on utilise /etc/hosts pour :

  • Simuler un DNS local sans avoir à configurer un serveur DNS.
  • Définir des alias pour les services (ex: mongo-config-r1).
  • Faciliter les tests en environnement isolé.

Dans un SI d'organisation, on utiliserait plutôt :

  • Un serveur DNS interne (comme Bind ou PowerDNS).
  • Des enregistrements CNAME pour les alias.
  • Un système de gestion de configuration (comme Ansible ou Puppet) pour déployer les configurations.
Question : Pourquoi voudrait-on désigner les services par des alias plutôt que par leur IP ou le nom du serveur qui les porte ?
Réponse :

Utiliser des alias présente plusieurs avantages :

  • Abstraction : Les alias masquent les détails d'implémentation (IP, nom d'hôte).
  • Flexibilité : On peut changer l'IP ou le serveur sous-jacent sans modifier les configurations des clients.
  • Lisibilité : Les noms comme mongo-config-r1 sont plus explicites que des IP.
  • Gestion centralisée : Les alias peuvent être gérés centralement via DNS ou /etc/hosts.
Question : Pourquoi la commande ss renvoie-t-elle plus d'informations (une colonne en plus) que lors de ses appels précédents ?
Réponse :

La commande ss -lntp (ou ss -Intp) affiche les sockets en écoute avec des informations supplémentaires, notamment :

  • La colonne Process (ajoutée avec -p) montre le PID et le nom du processus utilisant chaque socket.
  • Les options -l (listening), -n (numeric, pas de résolution DNS), -t (TCP), et -p (process) combinées donnent une vue détaillée des connexions actives et des processus associés.

3.2.6. Visualisation avec Kibana

Question : Pourquoi la visualisation est-elle par défaut sur les 15 dernières minutes ? Comment modifier cette période ?
Réponse :

Kibana utilise un filtre temporel par défaut de 15 minutes pour :

  • Se concentrer sur les données récentes (pertinentes pour le monitoring en temps réel).
  • Éviter de surcharger l'interface avec trop de données.

Pour modifier cette période :

  1. Dans l'interface Kibana, allez dans Discover ou un dashboard.
  2. En haut à droite, cliquez sur le sélecteur de temps (ex: "Last 15 minutes").
  3. Choisissez une période prédéfinie (ex: "Last 1 hour", "Last 24 hours") ou définissez une période personnalisée.
Question : Quelles visualisations vous sembleraient utiles pour analyser les logs Apache ?
Réponse :

Pour analyser les logs Apache, les visualisations suivantes sont utiles :

  • Histogramme des codes HTTP : Pour identifier les erreurs (404, 500, etc.).
  • Répartition des requêtes par URL : Pour voir quelles pages sont les plus visitées.
  • Nombre de requêtes par client (IP) : Pour détecter des activités suspectes (ex: attaques DDoS).
  • Temps de réponse moyen par URL : Pour identifier les pages lentes.
  • Répartition des user-agents : Pour connaître les navigateurs ou robots accédant au site.
  • Cartographie géographique des clients : Pour visualiser la provenance géographique des requêtes.
  • Évolution du trafic dans le temps : Pour identifier les pics de trafic.

3.2.7. Import de Données dans Elasticsearch

Question : À quoi sert l'argument --dryRun dans la commande mongorestore ?
Réponse :

L'option --dryRun permet de :

  • Simuler une restauration sans réellement importer les données.
  • Vérifier que la commande est correcte et que les données peuvent être lues sans erreur.
  • Estimer le temps et les ressources nécessaires pour la restauration réelle.
Question : À quoi sert la commande time ? Pourquoi l'utiliser ici ?
Réponse :

La commande time mesure le temps d'exécution d'une commande. Elle est utile ici pour :

  • Évaluer les performances de l'import des données.
  • Comparer différents scénarios (ex: avec/sans index).
  • Identifier des goulots d'étranglement (ex: temps d'indexation élevé).

La sortie de time inclut :

  • real : Temps total écoulé (incluant l'attente des E/S).
  • user : Temps CPU passé en mode utilisateur.
  • sys : Temps CPU passé en mode noyau.
Question : Pourquoi n'est-il pas possible de faire le tri côté client si le nombre de documents est important ?
Réponse :

Pour de grands volumes de données, le tri côté client est impossible car :

  • Le client devrait télécharger tous les documents avant de pouvoir les trier, ce qui est inefficace et coûteux en bande passante.
  • Les performances seraient dégradées par le transfert et le traitement local.
  • Elasticsearch est optimisé pour effectuer des tris côté serveur, en utilisant des index et des algorithmes efficaces.
Question : Qu'est-ce que le principe de pagination pour Elasticsearch ?
Réponse :

La pagination permet de diviser les résultats d'une requête en plusieurs pages, afin de :

  • Éviter de surcharger le client avec trop de données en une seule fois.
  • Améliorer les performances et la réactivité de l'application.

Elasticsearch utilise deux mécanismes principaux pour la pagination :

  • from et size :
    • from : Indice du premier document à retourner.
    • size : Nombre de documents par page.
    • Exemple : {"from": 10, "size": 20} retourne les documents 11 à 30.

    Limite : Par défaut, Elasticsearch limite from + size à 10 000. Au-delà, il faut utiliser search_after.

  • search_after :
    • Utilise le dernier document de la page précédente pour obtenir la page suivante.
    • Plus efficace pour les grands jeux de données, car il évite de recalculer les scores pour tous les documents.
    • Nécessite un champ de tri unique (comme _id ou un timestamp).

4. Réponses aux Questions des TP

4.1. Questions Générales sur NoSQL

Question : Quels sont les avantages et inconvénients de BASE par rapport à ACID ?
Réponse :

Avantages de BASE :

  • Scalabilité : Meilleure adaptation aux systèmes distribués et aux grands volumes de données.
  • Disponibilité : Priorité à la disponibilité même en cas de partition réseau.
  • Performance : Moins de surcharge liée aux transactions ACID.

Inconvénients de BASE :

  • Cohérence eventualle : Les données peuvent être temporairement incohérentes.
  • Complexité applicative : L'application doit gérer les incohérences.
  • Pas de rollback : Difficile d'annuler une opération en cas d'erreur.
Question : Quels sont les cas d'usage typiques pour NoSQL ?
Réponse :

NoSQL est particulièrement adapté pour :

  • Big Data : Stockage et traitement de grands volumes de données non structurées.
  • Applications web modernes : Gestion de données JSON (ex: profils utilisateurs, sessions).
  • IoT : Collecte et analyse de données provenants de capteurs.
  • Logs et analytics : Stockage et analyse de logs (ex: avec ELK).
  • Applications nécessitant une scalabilité horizontale : Ex: réseaux sociaux, jeux en ligne.

4.2. Questions sur MongoDB

Question : Pourquoi MongoDB utilise-t-il des documents JSON plutôt que des tables relationnelles ?
Réponse :

MongoDB utilise des documents JSON pour :

  • Flexibilité : Pas de schéma fixe, les champs peuvent varier d'un document à l'autre.
  • Correspondance naturelle avec les objets applicatifs : Les données sont souvent déjà sous forme JSON dans les applications modernes.
  • Performance : Moins de jointures nécessaires (les données liées sont souvent embarquées dans le même document).
  • Scalabilité : Les documents sont plus faciles à répartir sur plusieurs nœuds (sharding).
Question : Qu'est-ce qu'un replica set dans MongoDB ?
Réponse :

Un replica set est un groupe de serveurs MongoDB qui maintiennent les mêmes données, offrant :

  • Haute disponibilité : Si le nœud primaire tombe en panne, un secondaire est élu comme nouveau primaire.
  • Tolérance aux pannes : Les données sont répliquées sur plusieurs nœuds.
  • Lecture scalable : Les requêtes de lecture peuvent être réparties sur les nœuds secondaires.

Un replica set est composé de :

  • Un nœud primaire (reçoit les écritures).
  • Un ou plusieurs nœuds secondaires (répliquent les données).
  • Éventuellement un arbitre (participe aux élections sans stocker de données).
Question : Qu'est-ce que le sharding dans MongoDB ?
Réponse :

Le sharding est une technique de partitionnement horizontal qui permet de :

  • Répartir les données d'une collection sur plusieurs machines (shards).
  • Améliorer les performances en parallélisant les requêtes.
  • Augmenter la capacité de stockage en ajoutant des shards.

Composants clés :

  • Shards : Stockent une partie des données.
  • Config servers : Stockent les métadonnées du cluster.
  • Mongos : Routeur qui achemine les requêtes vers les shards appropriés.
Question : Comment choisir une clé de sharding dans MongoDB ?
Réponse :

Le choix de la clé de sharding est crucial pour les performances. Voici les critères à considérer :

  • Cardinalité élevée : La clé doit avoir beaucoup de valeurs distinctes pour éviter les "hotspots".
  • Distribution uniforme : Les requêtes doivent être réparties équitablement entre les shards.
  • Stabilité : La clé ne doit pas changer fréquemment (pour éviter les migrations de chunks).
  • Fréquence d'utilisation : La clé doit être souvent utilisée dans les requêtes.

Stratégies courantes :

  • Hachage : Bonne distribution, mais moins efficace pour les requêtes par plage.
  • Plage : Efficace pour les requêtes par plage, mais peut entraîner des déséquilibres.
  • Clé composée : Combine plusieurs champs pour optimiser la distribution et les requêtes.

4.3. Questions sur la Stack ELK

Question : Quelles sont les différences entre Logstash et Beats ?
Réponse :

Logstash :

  • Outil lourd avec une JVM.
  • Supporte des transformations complexes (filtrage, parsing, enrichissement).
  • Utilisé pour des pipelines avancés (ex: agrégation de multiples sources).

Beats :

  • Agents légers écrits en Go.
  • Spécialisés pour des tâches simples (ex: collecte de logs avec Filebeat).
  • Moins gourmands en ressources, idéaux pour les environnements embarqués.

Complémentarité :

  • Les Beats collectent les données et les envoient à Logstash pour traitement, ou directement à Elasticsearch.
  • Logstash est utilisé lorsque des transformations complexes sont nécessaires.
Question : Comment Elasticsearch gère-t-il la réplication et la tolérance aux pannes ?
Réponse :

Elasticsearch utilise les mécanismes suivants :

  • Réplication :
    • Chaque index est divisé en shards (partitions).
    • Chaque shard peut avoir une ou plusieurs répliques.
    • Les répliques sont stockées sur différents nœuds pour la redondance.
  • Tolérance aux pannes :
    • Si un nœud tombe en panne, ses shards primaires sont promus depuis les répliques.
    • Le cluster reste disponible tant qu'il y a suffisamment de répliques.
  • Détection des pannes :
    • Les nœuds s'envoient régulièrement des heartbeats.
    • Un nœud est considéré comme défaillant après un timeout.
Question : Quels sont les avantages de Kibana pour la visualisation des données ?
Réponse :

Kibana offre plusieurs avantages pour la visualisation :

  • Interface intuitive : Permet de créer des dashboards sans coder.
  • Intégration native avec Elasticsearch : Accès direct aux données indexées.
  • Visualisations variées : Graphiques, cartes, tables, métriques, etc.
  • Partage et collaboration : Possibilité de partager des dashboards et des requêtes.
  • Exploration interactive : Filtrage et drill-down sur les données.
  • Alerting : Création d'alertes basées sur des seuils ou des patterns.

4.4. Questions sur les Performances et l'Optimisation

Question : Comment optimiser les performances d'Elasticsearch ?
Réponse :

Plusieurs stratégies permettent d'optimiser Elasticsearch :

  • Index et mappings :
    • Définir des mappings explicites pour éviter le mapping dynamique.
    • Utiliser des types de champs adaptés (ex: keyword pour les champs triables).
  • Sharding :
    • Choisir un nombre de shards adapté à la taille des données et au matériel.
    • Éviter les shards trop petits ou trop grands.
  • Réplication :
    • Ajuster le nombre de répliques en fonction des besoins en disponibilité et en performance de lecture.
  • Matériel :
    • Allouer suffisamment de RAM (pour la heap JVM et le cache du système de fichiers).
    • Utiliser des disques SSD pour les performances d'E/S.
  • Requêtes :
    • Utiliser des filtres plutôt que des requêtes pour les critères booléens.
    • Limiter le nombre de résultats avec size et utiliser la pagination.
  • Configuration JVM :
    • Ne pas dépasser 50% de la RAM totale pour la heap JVM (pour laisser de la place au cache du système de fichiers).
    • Éviter les tailles de heap supérieures à 32 Go (pour éviter les pauses longues du garbage collector).
Question : Quelles sont les bonnes pratiques pour la gestion des logs avec ELK ?
Réponse :

Voici quelques bonnes pratiques pour gérer les logs avec ELK :

  • Normalisation des logs :
    • Utiliser des formats standardisés (ex: JSON, syslog).
    • Appliquer des parsers (comme Grok dans Logstash) pour extraire des champs structurés.
  • Filtrage et enrichissement :
    • Filtrer les logs non pertinents dès la collecte (ex: avec Filebeat).
    • Enrichir les logs avec des métadonnées (ex: géolocalisation, informations sur l'hôte).
  • Indexation :
    • Utiliser des index patterns basés sur la date (ex: logs-2023-*-*).
    • Configurer des politiques de rétention (ILM) pour gérer la rotation des index.
  • Visualisation :
    • Créer des dashboards pour surveiller les métriques clés (erreurs, temps de réponse, etc.).
    • Configurer des alertes pour les anomalies.
  • Sécurité :
    • Activer l'authentification et le chiffrement (TLS).
    • Restreindre les accès avec des rôles (RBAC).
  • Performance :
    • Optimiser les requêtes avec des agrégations et des filtres.
    • Utiliser des index dédiés pour les logs fréquents (ex: logs Apache dans un index séparé).
